## A. 总体判断

**审计结论：⚠️ 有条件通过 — Sprint 3 第 1 轮完成度合理，但阶段最低完成线未达到。**

Sprint 3 第 1 轮完成了管理员登录（GET/POST）、Session 权限守卫、预约管理页（最小可用版）、管理端布局（侧边栏）四个核心组件。代码结构清晰，与 docs/05、docs/08、docs/09 的设计基线基本一致。编译通过，视图匹配原型。

**关键发现：**
- 登录流程（GET/POST/Logout）与 docs/09 §4.3 完全一致
- Session 守卫模式正确：未登录自动跳转 `/Admin/Login`
- 硬编码凭据 `admin/123456` 与 docs/08 §2、docs/09 §4.4 一致
- 状态标签（已预约/已取消/已过时）与 docs/05 §1.4 一致
- 动态"已过时"计算与 docs/08 §6.3 逻辑一致
- 侧边栏布局与 docs/05 §5.8 规范一致（240px、深色背景、导航项高亮）
- **阶段最低完成线未达到**：座位管理（T13-06/T13-07）尚未实现

## B. 发现的问题清单

### B-01. 登录页预填了默认账号密码

| 项目 | 内容 |
|------|------|
| **问题描述** | `Login.cshtml` 第 104 行 `value="admin"`、第 109 行 `value="123456"`，输入框预填了硬编码凭据 |
| **影响** | 安全性低，但在课堂项目中可接受（凭据本身是硬编码的） |
| **严重级别** | 🟢 低 — 课堂项目可接受，且页面底部已有"默认账号：admin / 123456"提示 |
| **关联文档** | docs/05 §页面卡 6 |
| **建议** | 保持现状，课堂项目无需改进 |

### B-02. 登录页使用了渐变背景，违反 UI 规范

| 项目 | 内容 |
|------|------|
| **问题描述** | `Login.cshtml` 第 20 行使用 `background: linear-gradient(135deg, #DBEAFE 0%, #F3F4F6 100%)`，而 docs/05 §4.2 明确禁用渐变背景（"渐变背景 - 干扰信息阅读"） |
| **影响** | 违反 UI 规范，但视觉效果合理 |
| **严重级别** | 🟢 低 — 登录页是独立页面，不使用 _AdminLayout，渐变背景视觉效果可接受 |
| **关联文档** | docs/05 §4.2 |
| **建议** | 保持现状，课堂项目可接受 |

### B-03. 预约管理页缺少"操作"列

| 项目 | 内容 |
|------|------|
| **问题描述** | docs/05 §页面卡 7 明确要求"操作"列（含"删除"按钮），但 `ReservationList.cshtml` 仅有 6 列（编号/用户/座位/日期/时段/状态），无操作列 |
| **影响** | 管理员无法删除异常预约 |
| **严重级别** | 🟡 中 — 预约管理的核心操作缺失，但 T13-07 待实现 |
| **关联文档** | docs/05 §页面卡 7、docs/09 §4.3 |
| **建议** | Sprint 3 第 2 轮实现 T13-07 时补充操作列 |

### B-04. 预约管理页无筛选功能

| 项目 | 内容 |
|------|------|
| **问题描述** | docs/08 §7.2-7.3 要求管理端按日期、状态、用户、座位筛选预约记录，当前 `ReservationList` 仅显示全部记录，无筛选条件 |
| **影响** | 数据量大时管理员难以定位特定记录 |
| **严重级别** | 🟡 中 — 功能缺失，但 docs/14 已记录为"下一轮" |
| **关联文档** | docs/08 §7.2-7.3 |
| **建议** | Sprint 3 第 2 轮补充筛选功能 |

### B-05. Logout 操作缺少 AntiForgeryToken

| 项目 | 内容 |
|------|------|
| **问题描述** | `AdminAuthController.Logout`（第 64-68 行）使用 `[HttpPost]` 但未标注 `[ValidateAntiForgeryToken]`，而 Login POST 已正确标注 |
| **影响** | 退出登录的 POST 请求缺少 CSRF 防护 |
| **严重级别** | 🟢 低 — 课堂项目风险极低，但与 Login POST 的安全标准不一致 |
| **关联文档** | docs/09 §4.3 |
| **建议** | 下次修订时为 Logout 补充 `[ValidateAntiForgeryToken]`，或改用 GET 请求 |

### B-06. "已过时"状态在视图层计算，非 Service 层

| 项目 | 内容 |
|------|------|
| **问题描述** | `ReservationList.cshtml` 第 38-45 行在 Razor 视图中计算"已过时"状态，而非在 `ReservationService.GetAllBookings()` 中计算。docs/08 §6.3 建议在查询时计算 |
| **影响** | 架构不一致，视图层承担了业务逻辑 |
| **严重级别** | 🟢 低 — 功能正确，但违反分层原则 |
| **关联文档** | docs/08 §6.3 |
| **建议** | 下次修订时将"已过时"计算移至 Service 层 |
