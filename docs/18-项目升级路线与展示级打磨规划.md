# 项目升级路线与展示级打磨规划

> 阶段 18 — 在已交付 v1.0-final 基础上的升级方向与优先级规划
> 执行时间：2026-07-14
> 性质：规划文档，不涉及代码实现

---

## 一、当前项目真实基线判断

### 1.1 已实现了什么

| 维度 | 现状 |
|------|------|
| 功能 | 10/13 功能点：用户端浏览（首页+列表+详情）、管理端登录+Session守卫、预约查看、座位增删改 |
| 页面 | 6/9 原型已对接为 Razor 视图，另有 2 个布局文件 |
| 测试 | Playwright 10/10 + curl 7/7 = 17/17 通过 |
| 文档 | 17 份正文 + 审计报告，覆盖立项→交付全流程 |
| 数据库 | SQLite Code First，自动建表+种子数据（8 座位+3 用户），零手动操作 |
| 架构 | MVC + Service 层 + DI，三层结构清晰 |
| 部署 | `dotnet restore → dotnet run` 即可运行，v1.0-final tag 已推送 |

### 1.2 还缺什么

| 缺口 | 影响程度 |
|------|----------|
| 预约提交（选座→选时段→提交） | **核心** — 用户无法真正预约座位，MVP 闭环断裂 |
| 我的预约（查看/取消） | **核心** — 用户无法管理自己的预约 |
| 统计页 | 中等 — 管理端缺少数据概览 |
| 用户端登录态 | 中等 — 学生端无 Session，无法识别用户 |
| 预约管理筛选/搜索 | 低 — 管理端预约列表仅全量展示 |
| 响应式测试 | 低 — 仅有 CSS 代码审查，无 viewport 验证 |

### 1.3 现在最像一个课堂项目的地方

1. **核心业务闭环没有跑通** — 预约系统不能预约，这是最明显的"课堂感"来源
2. **管理员密码硬编码** — `admin/123456` 写死在代码里，真实产品不会这样做
3. **用户端无登录态** — 学生无法"登录"，无法关联预约记录到具体用户
4. **前端无任何交互反馈** — 没有 loading 状态、toast 提示、表单验证前端反馈、确认弹窗
5. **预约列表为空** — 演示时管理端预约管理页永远是空表格，无法展示数据流转

### 1.4 现在最有潜力继续拔高的地方

1. **预约闭环一旦补上，整个系统立刻"活"起来** — 这是投入产出比最高的单点升级
2. **17 份文档 + 审计链已经是加分项** — 大多数实习生项目没有这种工程纪律
3. **Playwright 自动化测试已有基础** — 扩展测试用例比从零搭建容易得多
4. **EF Core Code First 架构可扩展** — 加表、加字段、加种子数据都很自然
5. **原型-代码-文档三者已有对应关系** — 继续补齐后，面试时可以展示"设计→实现→验证"的完整链路

## 三、最值得优先投入的升级项清单

### 升级项 1：预约提交 + 我的预约 + 取消（Sprint 2 补全）

**当前问题表现：** 用户选完座位后无处可去，系统叫"预约系统"但不能预约。

**为什么会拉低项目成熟度：** 这是项目名称承诺的核心能力。缺少它，整个系统只是一个"座位展示站"。

**对面试展示的价值：** 补上后可以完整演示"选座→选时段→提交→查看→取消"闭环，这是最能证明"我做了一个真实系统"的证据。

**对接近商业化的价值：** 这是从"原型"到"可用产品"的分水岭。

**AI 可参与的工作点：**
- 根据 `docs/09-关键链路详细设计.md` 生成 `ReservationController` 和视图
- 根据 `prototype/static-v1/booking-submit.html` 和 `my-bookings.html` 生成 Razor 视图
- 扩展 `ReservationService` 的 `CreateReservation` 方法（冲突检测逻辑已有接口定义）
- 生成对应的 Playwright 测试用例

**建议开发步骤：**
1. 实现 `ReservationService.CreateReservation`（时段冲突检测 + 写入数据库）
2. 实现 `ReservationController.Create` GET（日期选择器 + 时段列表）
3. 实现 `ReservationController.Create` POST（提交预约 → 跳转我的预约）
4. 实现 `ReservationController.My` + 视图（预约列表 + 取消按钮）
5. 实现取消预约（确认弹窗 + 状态更新）
6. 补充 Playwright 端到端测试（预约→查看→取消全流程）

**关键产物：**
- `Controllers/ReservationController.cs`
- `Views/Reservation/Create.cshtml`、`My.cshtml`
- 扩展 `ReservationService.cs`
- 新增 Playwright 测试用例

**最低验收线：** 用户能选座→选时段→提交预约→在"我的预约"中看到→取消预约。管理端预约列表同步显示。

### 升级项 2：用户端登录态 + 预约关联

**当前问题表现：** 学生端无 Session，预约记录无法关联到具体用户。

**为什么会拉低项目成熟度：** 真实的预约系统一定有用户身份。无登录态意味着预约数据没有归属。

**对面试展示的价值：** 登录→预约→我的预约 是最自然的用户操作链路，面试官一看就懂。

**对接近商业化的价值：** 用户身份是所有后续功能（通知、权限、数据统计）的基础。

**AI 可参与的工作点：**
- 基于现有 `AdminAuthService` 的 Session 模式，为学生端创建 `StudentAuthService`
- 在 `HomeController` 添加登录/退出逻辑
- 在预约提交时写入 `UserId`

**建议开发步骤：**
1. 创建 `StudentAuthService`（基于 DemoUser 表，user1/user2/user3 登录）
2. 在 `_Layout.cshtml` 添加学生端登录/退出入口
3. 预约提交时自动写入当前登录用户的 UserId
4. "我的预约"只显示当前用户的记录

**关键产物：**
- `Services/StudentAuthService.cs`
- `Views/Home/Index.cshtml`（添加登录表单或用户选择）
- 修改 `ReservationController`（Session 鉴权）

**最低验收线：** 学生可以选择身份登录（user1/user2/user3），预约记录自动关联到登录用户，"我的预约"只显示自己的记录。

### 升级项 3：管理端统计页

**当前问题表现：** 管理端侧边栏没有统计入口，管理端缺少数据概览。

**对面试展示的价值：** 统计页是最容易展示"我懂数据分析"的页面，几个数字卡片就能让系统看起来更完整。

**对接近商业化的价值：** 管理决策依赖数据，统计页是管理端的标配。

**AI 可参与的工作点：**
- 根据 `prototype/static-v1/admin-stats.html` 生成 Razor 视图
- 实现 `AdminController.Stats`（查询座位数、预约数、今日预约数等）
- 在 `_AdminLayout.cshtml` 恢复统计页导航链接

**建议开发步骤：**
1. 实现 `AdminController.Stats` action（查询聚合数据）
2. 根据原型 `admin-stats.html` 生成 `Views/Admin/Stats.cshtml`
3. 在 `_AdminLayout.cshtml` 添加统计页导航链接
4. 补充测试

**关键产物：**
- 修改 `AdminController.cs`（新增 Stats action）
- `Views/Admin/Stats.cshtml`
- 修改 `_AdminLayout.cshtml`

**最低验收线：** 管理端侧边栏有"统计"入口，点击后显示座位总数、已预约数、可用数等统计卡片。

### 升级项 4：前端交互体验提升

**当前问题表现：** 操作后无任何反馈——新增座位后无提示，删除后无确认，表单提交后无 loading 状态。

**为什么会拉低项目成熟度：** 没有交互反馈的页面像一个静态 HTML 演示，不像一个真实产品。

**对面试展示的价值：** 交互细节是面试官判断"这个学生有没有产品感"的关键信号。

**对接近商业化的价值：** 用户体验是产品竞争力的核心。

**AI 可参与的工作点：**
- 在 `_Layout.cshtml` 添加 Bootstrap Toast 组件
- 在表单提交后添加前端验证反馈
- 在删除操作前添加确认弹窗
- 添加简单的 loading 状态

**建议开发步骤：**
1. 在 `_Layout.cshtml` 添加 Bootstrap Toast 容器
2. 在 Controller 中通过 `TempData` 传递操作结果消息
3. 在视图中读取 TempData 并显示 Toast
4. 在删除操作前添加 JavaScript 确认弹窗
5. 在表单提交按钮添加 loading 状态（disabled + spinner）

**关键产物：**
- 修改 `_Layout.cshtml`（Toast 容器）
- 修改各 Controller（TempData 消息）
- 修改各视图（Toast 显示 + 确认弹窗 + loading 状态）

**最低验收线：** 新增/编辑/删除座位后有绿色成功提示，删除前有确认弹窗，表单提交时按钮显示 loading。

**对面试展示的价值：** 面试官看到完整的测试套件 + CI 流水线，会认为这个学生有工程素养。

**对接近商业化的价值：** 自动化测试是持续交付的基础。

**AI 可参与的工作点：**
- 扩展 Playwright 测试用例（预约全流程、错误场景、权限边界）
- 生成 GitHub Actions CI 配置
- 添加测试覆盖率报告

**建议开发步骤：**

2. 新增错误场景测试（重复预约、超时段、未登录提交）

**关键产物：**
- 扩展 `tests/smoke.spec.ts`
- 新增 `.github/workflows/ci.yml`
- README 添加 CI badge

**最低验收线：** Playwright 测试覆盖预约全流程 + 至少 3 个错误场景，GitHub Actions CI 绿色通过。

### 升级项 6：Docker 一键部署 + 在线演示

**当前问题表现：** 只能 `dotnet run` 本地运行，面试官无法直接访问效果。

**为什么会拉低项目成熟度：** "你可以 clone 下来自己跑" 和 "这是在线地址，直接看" 的展示效果差距巨大。

**对面试展示的价值：** 面试官打开链接就能看到效果，比任何 PPT 都有说服力。

**对接近商业化的价值：** 容器化部署是现代开发的基本能力。

**AI 可参与的工作点：**
- 生成 `Dockerfile` 和 `docker-compose.yml`
- 生成部署说明文档
- 配置 GitHub Container Registry 或 Vercel/Railway 部署

**建议开发步骤：**
1. 创建 `Dockerfile`（多阶段构建：dotnet build → 运行）
2. 创建 `docker-compose.yml`（单容器部署）
3. 本地测试 `docker build && docker run`
4. 推送到 GitHub Container Registry
5. 或使用 Vercel/Railway 一键部署（更简单）
6. 在 README 添加在线演示链接

**关键产物：**
- `Dockerfile`
- `docker-compose.yml`
- `docs/19-部署说明.md`（可选）
- README 添加在线演示链接

**最低验收线：** `docker build && docker run` 后应用可访问，或有一个在线可访问的部署地址。

### 升级项 7：管理员密码数据库存储 + 学生端 Session 安全

**当前问题表现：** 管理员密码硬编码在代码中，学生端无 Session 超时保护。

**为什么会拉低项目成熟度：** 硬编码密码是安全反模式，面试官一看就知道"这个学生没考虑过安全"。

**对面试展示的价值：** 主动提到"我知道这里需要改进"比被面试官发现要好得多。

**对接近商业化的价值：** 安全是上线前的硬性要求。

**AI 可参与的工作点：**
- 将管理员账号迁移到数据库
- 在 `AdminAuthService` 中改为数据库查询 + 密码哈希
- 添加学生端 Session 超时保护

**建议开发步骤：**
1. 在 `AdminUsers` 表存储管理员账号（密码用 BCrypt 哈希）
2. 修改 `AdminAuthService.ValidateCredentials` 为数据库查询
3. 在学生端添加 Session 超时保护
4. 更新种子数据（添加管理员账号到数据库）

**关键产物：**
- 修改 `AppDbContext.cs`（新增 AdminUser 实体）
- 修改 `AdminAuthService.cs`（数据库查询 + 密码哈希）
- 添加 NuGet 包 `BCrypt.Net-Next`

**最低验收线：** 管理员密码不再硬编码在代码中，数据库中有加密存储的管理员账号。

## 四、后续升级路线图

### 第一轮：展示级补强（面试能拿出手）

**目标：** 让面试官打开项目链接时，看到的是一个"能跑的完整系统"，而不是"半成品原型"。

**建议先后顺序：**

| 顺序 | 升级项 | 理由 |
|------|--------|------|
| 1 | 预约闭环（升级项 1） | 最大短板，补上后系统立刻"活"起来 |
| 2 | 用户端登录态（升级项 2） | 预约需要归属到用户，否则数据无意义 |
| 3 | 管理端统计页（升级项 3） | 补齐管理端最后一块拼图 |
| 4 | 前端交互反馈（升级项 4） | 细节决定"课堂感"还是"产品感" |
| 5 | 文档修复（升级项 8） | 低投入高回报，消除审计遗留 |

**关键产物：**
- 完整的预约流程（选座→选时段→提交→查看→取消）
- 学生端登录态
- 管理端统计页
- Toast 提示 + 确认弹窗
- 修复后的 README 和 docs/17

**最低验收线：** 面试官可以完整走通"学生登录→选座→预约→查看→取消→管理端看到预约记录→统计页有数据"全流程。

| 项目 | 为什么不适合现在做 |
|------|-------------------|
| 微服务拆分 | 单体 MVC 对课程项目完全够用，微服务是过度设计 |
| 分布式缓存（Redis） | SQLite 单机足够，引入 Redis 增加部署复杂度但无实际收益 |
| Kubernetes 部署 | 一个座位预约系统不需要 K8s，Docker Compose 已经足够 |
| 完整的 RBAC 权限系统 | 只有一个管理员角色，RBAC 是过度设计 |
| 实时 WebSocket 推送 | 座位状态变更频率极低，轮询或页面刷新完全够用 |
| 国际化（i18n） | 课程项目面向中文用户，国际化无实际需求 |
| 完整的日志收集系统（ELK） | 本地文件日志 + 控制台输出对课程项目足够 |

---

## 六、最终结论

### 如果只继续打磨一轮，第一优先级应该做什么

**预约提交 + 我的预约 + 取消（升级项 1）+ 用户端登录态（升级项 2）**

理由：
1. 这是项目名称"座位预约系统"承诺的核心能力，缺了它系统名不副实
2. 补上后，演示时管理端预约列表不再为空，数据流转可见
3. 面试官可以完整走通"登录→选座→预约→查看→取消"全流程
4. 技术实现难度适中，借助 AI 可以在 1-2 天内完成
5. 一旦完成，项目的"课堂感"立刻消失，变成一个"能用的系统"

### 最能让面试官感受到项目成熟度的 3 个证据

1. **"我做了一个完整的预约闭环"** — 面试官亲自走通"登录→选座→选时段→提交→查看→取消→管理端看到记录"，这比任何 PPT 都有说服力

2. **"我有 17 份文档和审计报告"** — 展示 docs/ 目录，说明从立项到交付每一步都有记录和审计，这在实习生项目中极其罕见

3. **"我有自动化测试，而且全部通过"** — 展示 `npx playwright test` 的执行结果，说明这个项目不是"写完就跑"，而是有质量保障的
