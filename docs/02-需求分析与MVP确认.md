# 需求分析与 MVP 确认：图书馆座位预约系统

---

## 1. 目标用户与核心场景

### 1.1 用户画像

**用户A：学生小王**
- 大二学生，期末复习期间需要长时间在图书馆学习
- 痛点：早上 7 点就要去图书馆排队占座，耽误睡眠时间
- 期望：提前在线上看好哪个座位空着，预约好直接去坐下
- 使用频率：考试季每天 1-2 次，平时偶尔使用

**用户B：管理员李老师**
- 图书馆值班老师，负责日常座位管理
- 痛点：线下登记预约信息繁琐，数据难以统计
- 期望：在线查看所有预约记录，快速处理异常情况
- 使用频率：每天查看 1-2 次

### 1.2 核心场景

| 场景编号 | 场景名称 | 触发条件 | 用户行为 | 期望结果 |
|----------|----------|----------|----------|----------|
| S1 | 快速预约座位 | 小王想预约明天上午的座位 | 浏览座位列表 → 选择可用座位 → 选择时间段 → 提交预约 | 预约成功，"我的预约"中显示记录 |
| S2 | 查看预约状态 | 小王想确认自己的预约 | 进入"我的预约"页面 | 看到预约记录，状态显示"已预约" |
| S3 | 取消预约 | 小王明天有事去不了 | 在"我的预约"中点击取消 | 预约取消，座位恢复可用 |
| S4 | 管理员查看所有预约 | 李老师想了解今天的预约情况 | 登录管理后台 → 预约管理 | 看到所有用户的预约记录列表 |
| S5 | 管理员管理座位 | 李老师想新增一个座位 | 座位管理 → 新增座位 → 填写信息 → 保存 | 新座位出现在用户端列表中 |

## 2. 用户最关心的问题

通过"课堂替代调研"（基于校园图书馆场景假设 + 同类预约系统参考），归纳出用户最关心的 5 个问题：

| 优先级 | 用户问题 | 对应功能 |
|--------|----------|----------|
| P0 | **能不能快速看到哪些座位有空？** | 座位列表（状态标签） |
| P0 | **能不能顺利预约到座位？** | 预约提交（选座+选时间） |
| P1 | **能不能看到自己的预约记录？** | 我的预约（记录列表） |
| P1 | **预约后去不了，能不能取消？** | 取消预约 |
| P2 | **座位信息对不对？有没有设备？** | 座位详情 |

> 核心结论：**"看到空位 → 预约成功 → 查看记录"是用户最核心的三步，必须优先保证。**

## 4. 功能优先级矩阵

采用四级分层：**核心基础 → 主体业务 → 体验提升 → 暂缓**

### 4.1 优先级定义

| 级别 | 定义 | 处理方式 |
|------|------|----------|
| **核心基础** | 没有它系统无法运行 | 必须做，MVP 最低底线 |
| **主体业务** | 用户能完成核心任务，体验基本完整 | 必须做，是 MVP 的主体 |
| **体验提升** | 让系统更好用，但不影响核心功能 | 有时间就做，没时间可砍 |
| **暂缓** | 本期不做，留给未来版本 | 明确不做，不在本期范围内 |

### 4.2 功能优先级分配

| 编号 | 功能名称 | 优先级 | 理由 |
|------|----------|--------|------|
| U01 | 用户首页 | 核心基础 | 系统入口，没有首页无法开始 |
| U02 | 座位列表 | 核心基础 | 用户看到空位的唯一途径 |
| U04 | 预约提交 | 核心基础 | 系统核心功能，必须完成 |
| U05 | 我的预约 | 核心基础 | 用户查看自己预约的唯一途径 |
| U03 | 座位详情 | 主体业务 | 完善选座体验，但不是必须 |
| U06 | 取消预约 | 主体业务 | 完善预约流程，但不是必须 |
| A01 | 管理员登录 | 主体业务 | 管理端入口 |
| A02 | 预约管理 | 主体业务 | 管理员核心操作 |
| A03 | 座位管理 | 主体业务 | 管理员核心操作 |
| U08 | 预约时间选择优化 | 体验提升 | 减少用户选错时间的概率 |
| U09 | 预约提醒 | 体验提升 | 提升用户体验 |
| A04 | 统计页 | 体验提升 | 锦上添花，不是必须 |
| U07 | 座位搜索 | 暂缓 | 座位数量少，不需要搜索 |
| A05 | 预约数据导出 | 暂缓 | 课堂项目不需要导出 |
| A06 | 座位状态批量操作 | 暂缓 | 座位数量少，不需要批量操作 |

### 4.3 优先级分布图

```
核心基础（4个）：U01, U02, U04, U05
主体业务（5个）：U03, U06, A01, A02, A03
体验提升（3个）：U08, U09, A04
暂缓（3个）：U07, A05, A06
```

## 5. MVP 范围确认

### 5.1 MVP 定义

**MVP（Minimum Viable Product）** = 核心基础 + 主体业务，共 **9 个功能**。

这 9 个功能完成后，系统可以：
- 学生用户：浏览座位 → 选择座位 → 提交预约 → 查看预约 → 取消预约
- 管理员：登录 → 管理座位 → 管理预约

### 5.2 MVP 功能清单

| 优先级 | 编号 | 功能 | 页面 | 工作量估算 |
|--------|------|------|------|------------|
| 核心基础 | U01 | 用户首页 | `/Home/Index` | 小 |
| 核心基础 | U02 | 座位列表 | `/Seat/List` | 中 |
| 核心基础 | U04 | 预约提交 | `/Reservation/Create` | 中 |
| 核心基础 | U05 | 我的预约 | `/Reservation/My` | 小 |
| 主体业务 | U03 | 座位详情 | `/Seat/Detail/{id}` | 小 |
| 主体业务 | A01 | 管理员登录 | `/Admin/Login` | 小 |
| 主体业务 | A02 | 预约管理 | `/Admin/ReservationList` | 中 |
| 主体业务 | A03 | 座位管理 | `/Admin/SeatList` | 中 |

**总计：9 个功能，9 个页面**

> 说明：取消预约在"我的预约"页内通过模态框操作；座位编辑在"座位管理"页内通过模态框操作，不单独设页。

### 5.3 MVP 页面清单

**用户端页面（5个）：**

| 页面 | 路由 | 功能 | 页面类型 |
|------|------|------|----------|
| 用户首页 | `/Home/Index` | 欢迎信息、快捷入口 | 静态展示 |
| 座位列表 | `/Seat/List` | 座位卡片列表，状态标签 | 列表页 |
| 座位详情 | `/Seat/Detail/{id}` | 座位详细信息 + 预约按钮 | 详情页 |
| 预约提交 | `/Reservation/Create` | 选座+选时间+确认提交 | 表单页 |
| 我的预约 | `/Reservation/My` | 预约记录列表 + 取消预约（模态框） | 列表页 |

**管理端页面（4个）：**

| 页面 | 路由 | 功能 | 页面类型 |
|------|------|------|----------|
| 管理员登录 | `/Admin/Login` | 登录表单 | 表单页 |
| 预约管理 | `/Admin/ReservationList` | 所有预约记录表格 | 列表页 |
| 座位管理 | `/Admin/SeatList` | 座位列表 + 增删改（新增/编辑使用模态框） | 列表页 |
| 统计页 | `/Admin/Statistics` | 预约统计数据 | 展示页 |

> 统计页降级为简单表格展示，不使用图表库。统计页已纳入本期 MVP 开发范围。

### 5.4 体验提升功能处理

以下 3 个体验提升功能，**有时间就做，没时间可砍**：

| 编号 | 功能 | 处理方式 |
|------|------|----------|
| U08 | 预约时间选择优化 | 在预约提交页面增加"该时段是否已被预约"的提示 |
| U09 | 预约提醒 | 预约成功后用 Bootstrap Alert 显示成功提示 |
| A04 | 统计页 | 用简单表格展示，不做图表 |

## 6. 本期明确不做的功能

| 编号 | 功能 | 原因 |
|------|------|------|
| U07 | 座位搜索 | 座位数量少（预设 10-20 个），不需要搜索功能 |
| A05 | 预约数据导出 | 课堂项目不需要导出功能 |
| A06 | 座位状态批量操作 | 座位数量少，逐个操作即可 |
| - | 用户注册/登录 | 用户端用体验账号切换，不做注册体系 |
| - | 在线支付 | 图书馆座位免费，不需要支付 |
| - | 扫码签到 | 需要硬件配合，课堂项目无法实现 |
| - | 消息推送 | 需要第三方服务，本期用页面提示替代 |
| - | 多馆区联动 | 只做单馆区，逻辑简单 |
| - | 座位实时状态推送 | 不做 WebSocket，刷新页面获取最新状态 |
| - | 移动端适配 | 以 PC 端演示为主 |

## 7. 需求评审结论

### 7.1 保留项（MVP 必做）

| 序号 | 功能 | 理由 |
|------|------|------|
| 1 | 用户首页 | 系统入口，必须保留 |
| 2 | 座位列表 | 用户看到空位的唯一途径 |
| 3 | 座位详情 | 完善选座体验 |
| 4 | 预约提交 | 系统核心功能 |
| 5 | 我的预约 | 用户查看预约的唯一途径 |
| 6 | 取消预约 | 完善预约流程 |
| 7 | 管理员登录 | 管理端入口 |
| 8 | 预约管理 | 管理员核心操作 |
| 9 | 座位管理 | 管理员核心操作 |

### 7.2 删减项（从候选池中移除）

| 序号 | 功能 | 删减理由 |
|------|------|----------|
| 1 | 座位搜索 | 座位数量少，搜索价值不大 |
| 2 | 预约数据导出 | 课堂项目不需要 |
| 3 | 座位状态批量操作 | 座位数量少，不需要批量操作 |

### 7.3 暂缓项（体验提升，有时间再做）

| 序号 | 功能 | 暂缓理由 |
|------|------|----------|
| 1 | 预约时间选择优化 | 可以在基础版本上迭代 |
| 2 | 预约提醒 | 用简单 Alert 替代即可 |
| 3 | 统计页（图表版） | 先用表格，有时间再加图表 |

### 7.4 评审结论

**本轮需求分析共梳理候选功能 15 个，最终确认 MVP 范围 9 个功能、9 个页面。**

- 保留项：9 个（核心基础 4 个 + 主体业务 5 个）
- 删减项：3 个（从候选池中移除）
- 暂缓项：3 个（体验提升，有时间再做）

**MVP 范围可在 2-3 周内由单人完成，符合课堂项目要求。**

## 8. 进入 PRD Lite 前的最终口径

### 8.1 项目边界确认

| 维度 | 确认内容 |
|------|----------|
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 用户端 | 6 个页面，体验账号切换 |
| 管理端 | 5 个页面，Session/Cookie 登录 |
| 总页面数 | 11 个 |
| 总功能数 | 9 个（MVP） |
| 开发周期 | 2-3 周 |
| 交付方式 | GitHub 或 Gitee 仓库 |

### 8.2 核心业务规则确认

| 规则 | 确认内容 |
|------|----------|
| 时间段粒度 | 1 小时，可选时段 8:00-21:00（共 13 个时段） |
| 座位状态判断 | 以当前日期+当前时间为基准，该时段已被预约则显示"已预约" |
| 取消预约规则 | 仅允许取消未到开始时间的预约，已过时间不可取消 |
| 预约冲突检测 | 同一座位同一时段不可重复预约 |
| 体验账号数量 | 3 个，通过 Seed Data 自动创建 |
| 管理员账号 | 1 个，硬编码在代码中 |

### 8.3 页面工作量估算

| 页面类型 | 页面数 | 单页工作量 | 总工作量 |
|----------|--------|------------|----------|
| 静态展示页 | 1 | 0.5 天 | 0.5 天 |
| 列表页 | 4 | 1 天 | 4 天 |
| 详情页 | 1 | 0.5 天 | 0.5 天 |
| 表单页 | 2 | 1 天 | 2 天 |
| 展示页（统计） | 1 | 1 天 | 1 天 |
| **合计** | **9** | - | **8 天** |

> 加上数据库设计、EF Core 配置、布局模板等工作，总工作量约 **10-12 个工作日**，符合 2-3 周工期。
