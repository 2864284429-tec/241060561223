# 项目立项单：图书馆座位预约系统

---

## 1. 项目基本信息

| 项目 | 内容 |
|------|------|
| 项目名称 | 图书馆座位预约系统（Library Seat Reservation System） |
| 项目类型 | 课程实践项目（模拟真实工作流程） |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor 视图 + SQL Server LocalDB + EF Core + Bootstrap |
| 开发环境 | Visual Studio 2022 / VS Code + .NET 8 SDK |
| 代码托管 | GitHub 或 Gitee 仓库 |
| 版本管理 | Git，关键节点自动 commit |

---

## 2. 项目背景

图书馆是大学里最抢手的资源之一。每到考试季，座位供不应求，现场排队、占座、纠纷时有发生。很多同学一大早就去图书馆"人肉占座"，既浪费时间，也容易产生矛盾。

本项目要解决的问题很简单：**让同学们在线上就能看到哪些座位还有空位，提前预约，到了图书馆直接坐下学习，不用再抢座。**

系统不做商业化运营，不做复杂账号体系，核心目标是在课堂范围内完整走一遍"从立项到交付"的软件开发流程。

---

## 3. 目标用户

| 用户角色 | 说明 |
|----------|------|
| **普通用户（学生）** | 需要预约图书馆座位的同学，通过体验账号登录即可使用全部用户端功能 |
| **管理员** | 负责管理座位信息和处理预约数据，通过轻量级登录进入管理后台 |

> 说明：本期不做注册/登录体系，用户端采用体验账号切换方式模拟；管理端采用 Session/Cookie 轻量登录。

---

## 4. 项目目标

### 4.1 核心目标

完成一个**可演示、可验收**的图书馆座位预约系统，覆盖"选座 → 预约 → 查看 → 取消"完整链路，以及管理端的座位与预约管理。

### 4.2 能力目标

通过本项目，熟悉以下开发技能：

- ASP.NET Core MVC 项目搭建与路由配置
- Razor 视图页面开发
- EF Core 数据库操作（Code First）
- Bootstrap 前端页面布局
- 简单的权限控制（Session/Cookie）
- Git 版本管理与 GitHub/Gitee 托管

### 用户端（学生使用）

| 序号 | 功能 | 说明 |
|------|------|------|
| 1 | 用户首页 | 展示欢迎信息、当前日期、座位概览入口 |
| 2 | 座位列表 | 展示所有座位，区分"可用"与"已预约"状态 |

### 管理端（管理员使用）

| 序号 | 功能 | 说明 |
|------|------|------|
| 1 | 管理员登录 | 轻量级 Session/Cookie 登录 |
| 2 | 预约管理 | 查看所有预约记录，可删除异常预约 |
| 3 | 座位管理 | 增删改查座位信息（名称、位置、容量、设备等） |

## 6. 本期暂不开发内容

| 序号 | 功能 | 说明（为什么不做） |
|------|------|---------------------|
| 1 | 用户注册/登录 | 用户端用体验账号切换即可，复杂账号体系超出本期范围 |
| 2 | 在线支付 | 图书馆座位免费使用，不需要支付 |
| 3 | 扫码签到 | 需要硬件配合，课堂项目无法实现 |
| 4 | 消息推送 | 需要第三方服务，本期用页面提示替代 |
| 5 | 多馆区联动 | 只做单馆区，逻辑简单、演示清晰 |
| 6 | 座位实时状态推送 | 不做 WebSocket 实时推送，刷新页面获取最新状态 |
| 7 | 移动端适配 | 以 PC 端演示为主，Bootstrap 自带基础响应式即可 |

## 7. 页面规模与页面分布

### 页面清单

**用户端页面（6个）：**

| 页面 | 路由建议 | 主要内容 |
|------|----------|----------|
| 用户首页 | `/Home/Index` | 欢迎信息、快捷入口、今日概览 |
| 座位列表 | `/Seat/List` | 座位卡片列表，显示状态标签 |
| 座位详情 | `/Seat/Detail/{id}` | 座位详细信息 + 预约按钮 |
| 预约提交 | `/Reservation/Create` | 选座 + 选时间 + 确认提交 |
| 我的预约 | `/Reservation/My` | 当前用户预约记录列表 |
| 取消预约 | `/Reservation/Cancel/{id}` | 确认取消 + 状态更新 |

**管理端页面（5个）：**

| 页面 | 路由建议 | 主要内容 |
|------|----------|----------|
| 管理员登录 | `/Admin/Login` | 登录表单 |
| 预约管理 | `/Admin/ReservationList` | 所有预约记录表格 |
| 座位管理 | `/Admin/SeatList` | 座位列表 + 增删改操作 |
| 座位编辑 | `/Admin/SeatEdit/{id}` | 座位信息编辑表单 |
| 统计页 | `/Admin/Statistics` | 预约统计图表 |

**总计：11 个页面**，规模适中，一个人在 1-2 周内可完成。

## 8. 关键业务主流程

### 用户预约流程

```
用户首页 → 座位列表 → 选择可用座位 → 座位详情 → 点击预约
    → 选择日期和时间段 → 确认提交 → 预约成功
    → 在"我的预约"中查看记录
```

### 用户取消流程

```
我的预约 → 找到要取消的记录 → 点击取消 → 确认弹窗
    → 确认取消 → 状态更新为"已取消" → 座位恢复可用
```

### 管理员管理流程

```
管理员登录 → 进入管理后台
    → 座位管理：查看列表 → 新增/编辑/删除座位
    → 预约管理：查看所有预约 → 删除异常预约
    → 统计页：查看预约总数、使用率等
```

## 9. 成功标准

以下是项目完成时**必须能够演示**的操作清单：

| 序号 | 演示动作 | 验收标准 |
|------|----------|----------|
| 1 | 打开用户首页 | 页面正常显示，能看到座位概览信息 |
| 2 | 浏览座位列表 | 所有座位按状态显示（可用/已预约） |
| 3 | 点击某个座位查看详情 | 显示座位名称、位置、设备等信息 |
| 4 | 提交一个预约 | 选择座位和时间段，提交后提示成功 |
| 5 | 在"我的预约"中看到刚提交的记录 | 记录信息完整，状态显示"已预约" |
| 6 | 取消刚提交的预约 | 确认后状态变为"已取消"，座位恢复可用 |
| 7 | 用另一个座位再预约一次 | 预约列表中能看到两条记录 |
| 8 | 登录管理后台 | 输入管理员账号密码，成功进入 |
| 9 | 在座位管理中新增一个座位 | 新座位出现在用户端列表中 |
| 10 | 在座位管理中删除一个座位 | 座位从列表中消失 |
| 11 | 在预约管理中查看所有预约 | 能看到所有用户的预约记录 |
| 12 | 在统计页查看数据 | 显示预约总数、各座位使用次数等 |

> 每一条都是可操作、可截图、可录屏的动作，方便提交时作为验收依据。

## 10. 风险与假设

### 风险

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|----------|
| 时间不够用 | 高 | 功能做不完，演示效果差 | 严格控制功能范围，先做核心链路，再补管理端 |
| 页面数量超出预期 | 中 | 页面太多做不完 | 管理端页面共用布局模板，减少重复工作 |
| 数据库配置出问题 | 中 | EF Core 连接失败 | 用 LocalDB，出问题时参考官方文档或求助同学 |
| AI 辅助代码不稳定 | 中 | 生成的代码有 bug | 先理解代码逻辑再采用，不要盲目复制粘贴 |
| 一个人工作容易拖延 | 高 | 项目进度落后 | 设置每日小目标，每完成一个功能就 commit |
| 不确定的 UI 设计问题 | 中 | 页面丑或难用 | 用 Bootstrap 默认样式，不追求个性化设计 |

### 假设

| 假设 | 说明 |
|------|------|
| 开发环境已就绪 | Visual Studio 或 VS Code + .NET 8 SDK 已安装 |
| 数据库可用 | SQL Server LocalDB 可正常连接 |
| 网络通畅 | 能正常访问 GitHub/Gitee 进行代码托管 |
| 体验账号数量固定 | 用户端体验账号不超过 3 个，足够演示 |
| 管理员账号固定 | 1 个管理员账号，硬编码在代码中即可 |

## 11. 里程碑建议

| 里程碑 | 内容 | 预计周期 |
|--------|------|----------|
| **M1：项目搭建完成** | 创建项目、配置 EF Core 数据库、完成数据模型设计、用户端首页和座位列表页面 | 第 1 周 |
| **M2：核心功能可用** | 座位详情、预约提交、我的预约、取消预约——用户端完整链路走通 | 第 2 周 |

> 总工期约 **2-3 周**。M1-M3 是开发阶段，M4 是收尾阶段。如果时间紧张，M3 的统计页可以简化为基础表格，不需要图表。
